# Slop Quick

문답 없는 단일 패스 슬롭 제거. **가성비가 존재 이유다** - slop-fix 의 진단↔문답↔상수화 3단 왕복을 걷어내고, **기계적으로 확정된 것 + 레이아웃 두 문제**만 자동으로 쳐서 화면을 눈에 띄게 개선한다. 나머지(판단이 갈리는 것)는 손대지 않고 목록으로 넘긴다.

## slop-fix 와의 경계 (혼동 금지)

- **slop-quick (이 스킬)**: 메인 컨텍스트 단일 패스. slop-auditor 디스패치 없음, 계층 루프 없음, 문답 0. 확정 항목 자동 적용. 빠르고 얕다.
- **slop-fix**: 의도 인터뷰 + 4계층(콘텐츠→레이아웃→타이포→컬러) 문답 + 계층별 상수화. 느리고 깊다. 재설계·구조 재편이 필요할 때.
- **판단 규칙**: 이 스킬이 애매 버킷(hybrid·judgment·재설계·구조변경)으로 분류한 항목이 3개 이상 쌓이거나, 사용자가 "제대로 다시 보자" 면 slop-fix 로 승격 제안. 이 스킬은 slop-fix 를 대체하지 않고, 그 앞단의 저비용 통과다.

## 핵심 계약 (작성 시점 레드팀 - 아래 5개는 알려진 실패의 재발 차단선)

1. **헤드라인은 레이아웃 두 문제, AUTO 는 덤이다 (0건이 정상이다).** 이 스킬의 **정의된 산출물은 레이아웃 두 문제(여백·그리드)의 수정**이고, code-tell 자동 수정(그라디언트 교체·em-dash·글로우 삭제 등)은 **있으면 얹는 부수 처리**다. code-kind 검출기도 regex 휴리스틱이라 오탐이 흔하므로 **AUTO 버킷이 0건인 경우는 실패가 아니라 정상**이다 (드라이런 실증: 네오브루탈리즘·영어 네이티브 화면에서 code 히트 5개 전부 오탐이라 AUTO 0건이 정답이었다). **AUTO 가 비었다고 뭔가 지어내 자동수정하지 말 것** - 억지 자동수정은 의도된 미학을 파괴하는 오탐이다. 리포트는 반드시 "화면이 체감상 달라졌는가"를 별도로 판정해 노출한다 (before/after 렌더 가능하면 1쌍). **레이아웃을 빼먹고 code-tell 만 치거나, AUTO 0건인데 위생으로 분량 채워 "완료"로 파는 것이 이 스킬의 정의된 실패다.**
2. **여백을 눈대중으로 조절하지 않는다.** "여유 있는 레이아웃"은 개별 padding 을 손으로 키우는 것이 아니라 **선언된 간격 래더**(base + 내부<그룹<블록<섹션, 각 ≥1.5×)를 뽑아 전 섹션에 도출한다. 래더를 안전하게 뽑을 수 없는 화면이면 여백 수정은 **건너뛰고 propose 로 넘긴다** - 눈대중 조절(unpartitioned-space)의 재생산 금지.
3. **미러 교대를 그리드 완성으로 팔지 않는다.** 그리드 수정 성공 = 관련 섹션의 세로 분할선 x 가 **단일 목표값으로 수렴 + 앵커 섹션 존재**. 42/58·58/42 를 섹션마다 뒤집는 미러 교대는 x 가 두 값으로 갈라지는 눈속임이라 실패다.
4. **자동 적용은 스타일 전용, 의미는 절대 안 건드린다.** AUTO 버킷의 수정 범위 = 색 토큰·그라디언트/글로우/장식 삭제·radius 스케일·타이포 문자(em-dash 등)·간격 토큰. **카피·문구·정보 구조·섹션 경계는 자동 수정 대상이 아니다** - 콘텐츠 애매성은 무조건 propose. (문답이 없으므로 의미 결정을 자동화하면 통제 불능이다.)
5. **되돌릴 수 있게 적용한다.** 시작 전 git 상태 확인 - 더러운 트리면 작업 브랜치를 만들거나 사용자에게 커밋/스태시를 권한다. **수정 1건 = 커밋 1개** (한 항목 롤백이 가능해야 한다). 대상이 버전관리 밖이면 `<대상>-quick` 복사본에 git init 후 진행.

## 절차 (단일 패스 6스텝)

### 0. 준비 (30초)
- 대상 화면 경로·스택 파악. git 상태 확인 → 계약 5 대로 되돌리기 안전 확보.
- **과감도 기본값**: 이 스킬은 문답이 없으므로 과감도는 항상 **"표면 정리 + 선언된 레이아웃 수정"** 으로 고정이다. 재설계·구조 재편(섹션 통합/삭제)은 이 스킬의 권한 밖 - 발견되면 propose 로만 올린다. 브랜드 시그니처로 의심되는 요소(로고 연출·눈에 띄는 고유 장치)는 자동 삭제하지 않고 propose.

### 1. 스캔 = 체크리스트 (기계 검출)
- `node scripts/scan-slop-signals.mjs <대상경로> --json` 실행. 산출은 code/hybrid 검출기의 file:line 히트 + 각 항목 `detect.note`(오탐 조건).
- 이 JSON 이 체크리스트의 뼈대다. 무히트 검출기는 "확인했고 없음"으로 리포트에 기록(커버리지 증명).

### 2. 3-버킷 분류 (SSOT 런타임 판정 - 항목 수·임계값 하드코딩 금지, SSOT 에서 읽는다)
각 히트를 아래 게이트로 가른다:
- **AUTO (확정 → 자동 적용)**: 네 조건 **전부** 충족해야 한다 - (i) `detect.kind === 'code'` (ii) escape 가 기계적 치환 또는 장식-전용 삭제 (iii) **`detect.note` 의 오탐 조건을 히트별로 능동 통과** - 메인 컨텍스트가 1건씩 육안으로 "이 히트가 note 가 경고하는 오탐 상황이 **아님**"을 적극 확인한다 (전제 불성립·의도된 idiom·다른 패턴 오검출이면 통과 실패). 판단이 조금이라도 갈리면 통과 아님 → propose (iv) 수정이 계약 4 의 스타일 범위 안. 하나라도 어긋나면 AUTO 아님. **code-kind 라는 이유만으로 자동 적용 금지** (드라이런 실증: dead-cta 가 진짜 앵커를, untranslated-ui-terms 가 영어 네이티브 화면을, ubiquitous-soft-shadow 가 하드 오프셋 섀도를 오검출 - 전부 note 능동검증에서 걸러야 했다). AUTO 가 이렇게 0건이 되는 것은 정상이다(계약 1).
- **LAYOUT (확정 → 자동 적용, 아래 스텝 3)**: 여백·그리드 두 문제 전용. 택소노미 id 로는 `undisciplined-grid`(엇나간 분할선)·`unpartitioned-space`류(눈대중 여백)에 대응하지만, 이 스킬은 그 둘의 **경량 처방만** 집행한다.
- **PROPOSE (애매 → 자동 수정 안 함)**: 위 둘에 안 드는 전부 - `hybrid`·`judgment` 판단 항목, 재설계·foundation 결함, 섹션 통합/삭제 등 구조 변경, 카피·콘텐츠 애매성, escape 가 기계적 스타일 op 가 아닌 것. 스텝 5 의 옵트인 목록으로.

### 3. 레이아웃 두 문제 (이 스킬의 헤드라인 - 계약 2·3 준수)
렌더 가능하면 실측/판독으로, 불가하면 CSS·마크업 리터럴 실측으로:
- **(a) 여유 레이아웃**: 콘텐츠 대비 답답한 섹션을 찾되 개별 여백을 만지지 않는다. 화면의 간격 리터럴을 전수 수집 → base + 4티어(내부<그룹<블록<섹션, 각 ≥1.5×) **래더를 선언** → 전 섹션 간격을 래더에서 도출해 치환. 래더가 안전하게 안 뽑히면 이 항목 skip → propose.
- **(b) 관통 그리드**: 다열 섹션들의 세로 분할선 x 를 전수 측정 → 흩어지거나 미러 교대면 **단일 목표 x 로 수렴 + 앵커 섹션 지정** → 관련 섹션 재정렬. 성공 판정은 단일 x 수렴이지 미러 통일이 아니다.
- 삭제로 공간이 비면 남은 콘텐츠를 재중앙·재비율(장식 추가 금지, 기존 콘텐츠 재배치만) - void 방치 금지.

### 4. 적용 (자동, 커밋 단위)
- AUTO 버킷 + LAYOUT 두 처방을 적용. 각 수정 = 커밋 1개. 삭제는 관련 CSS·JS·orphan-ref(색-명명 클래스·미사용 var·키프레임)까지 함께.
- 스코프 밖(재설계·구조·카피) 신규 결정 금지. 확정 스타일 payload 만.

### 5. 요약 리포트 + 2차 제안
아래 고정 규격으로 사용자에게 보고한다:
```
## Slop Quick: <대상> <날짜>
### 자동 수정됨 (커밋 N개)
| # | 텔(SSOT id) | 처리(치환/삭제) | 파일 | 커밋 |
### 레이아웃
- 여백: <래더 선언값 또는 "skip - propose 로 이관"> 
- 그리드: <단일 x 수렴값·앵커 섹션 또는 "해당 없음">
### 체감 변화 판정 (계약 1)
<before/after 1쌍 경로 또는 diff 요약 + "3초 안에 달라 보이는가" 예/아니오>
### 2차 수정 제안 (애매 - 자동 수정 안 함, 옵트인)
| # | 항목(SSOT id) | 왜 자동 불가(hybrid/judgment/재설계/구조/카피) | 권장 경로(slop-fix / 스팟) |
```
- **2차 제안 목록에서 사용자가 항목을 고르면**: 그 항목이 스타일 스팟이면 이 스킬이 그 자리에서 좁게 처리, 재설계·구조·계층 확정이 필요하면 **slop-fix 로 승격**해 넘긴다.

## 흔한 함정 (이 스킬 전용)
- **애매한 걸 AUTO 로 밀지 말 것.** hybrid 히트를 "그냥 code 처럼 보이니까" 자동 적용하면 오탐(의도된 idiom 파괴)이 난다 - 게이트는 `detect.kind === 'code'` 실측이지 눈짐작이 아니다.
- **레이아웃을 빼먹고 code-tell 만 치지 말 것.** 그러면 화면이 안 바뀌고 "위생으로 분량 채우기"가 된다 (계약 1). 레이아웃 두 문제가 이 스킬의 체감 산출물이다.
- **문답으로 새지 말 것.** 이 스킬의 정체성은 문답 0 이다 - 애매하면 묻지 말고 propose 로 넘겨라. 물어야 할 만큼 판단이 갈리면 그건 slop-fix 의 일이다.
- **과감도를 넘지 말 것.** 재설계·섹션 통합/삭제·카피 재작성은 권한 밖 - 발견은 반갑지만 집행은 propose 후 slop-fix.
- em-dash(U+2014) 0건 (SSOT 원문 인용부 예외).
