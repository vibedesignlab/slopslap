# Slop Fix

대상 웹사이트(스택 불문: 이 레포, 형제 Next+MUI 프로젝트, 순수 HTML+CSS, Tailwind 등)의 AI-slop 과 디자인 층 결함을 고치는 작업 스킬. 역할 분리가 이 세트의 핵심이다:

- **점검 = `slop-auditor` 에이전트** (read-only, 독립 컨텍스트). 생성한 쪽이 자기 검증하지 않는다.
- **수정 = 이 스킬** (메인 컨텍스트에서 Edit). 에이전트의 층별 리포트를 승인받아 층 순서대로 적용한다.

핵심 원칙 (전 단계 적용):
- **환경 독립.** 이 세트는 스택이 아니라 구조를 고친다. 점검·수정 대상은 구조적 토큰(색 역할, 타입 스케일, 간격)과 개념적 컴포넌트(히어로·내비·카드·섹션)이고, MUI/순수 CSS/Tailwind 는 증거를 캐고 수정을 표현하는 문법일 뿐이다. 다른 스택 대상에선 그 프로젝트의 토큰 체계(CSS :root, tailwind config 등)를 찾아 그 문법으로 처방한다. 스캐너 스크립트가 그 스택에 안 맞으면 같은 신호 의도를 grep 으로 재현한다.
- **과감하게, 빼는 쪽으로.** AI스러움의 수정은 대부분 빼기다: 기믹 위젯 삭제, 카피 축소, 섹션 통합, 콘텐츠 뎁스 감소. 처방 사다리는 삭제 > 축소 > 교체이고, 색만 바꾸는 소심한 재스타일링은 실패 패턴이다. 과감해도 된다. 위험은 승인 게이트가 이미 막아준다.
- **위생 수정은 기본 스코프가 아니다.** px→rem, 토큰 정리, a11y 가드 같은 화면이 안 바뀌는 코드 위생은 사용자가 명시로 요청할 때만 진행한다. 이 스킬의 기본 산출물은 "화면이 덜 AI스러워 보이는" 수정이다. 위생만 잔뜩 고치고 화면이 그대로면 이 스킬은 실패한 것이다.
- **수정 순서는 의존성 순서다.** L0(콘텐츠 구조)을 고치면 그 요소의 L1~L3 발견이 무효가 될 수 있다. 반드시 L0 부터 적용하고, 잠정 태그가 붙은 하류 발견은 상류 수정 후 재점검한다.
- **수정은 반드시 사용자 승인 후.** 점검(에이전트 디스패치)은 자율, 수정(쓰기)은 승인 게이트 뒤.
- **처방의 출처는 지식 베이스 + 대상 토큰.** 무엇을 고칠지의 근거는 이 레포에 상주하는 지식(aiSlopTaxonomyData 의 escape/escapeNote, layoutTaxonomyData 아키타입, stable-layout 규약)에서 오고, 어떻게 표현할지는 대상 프로젝트 자신의 토큰 체계로 쓴다. 처방을 즉석에서 발명하지 않는다.
- em-dash(U+2014) 0건. 모든 산출물·수정 코드에서.

**왜 기계 검사로 회귀 검증하나 (6단계 근거):** 수정이 실제로 먹혔는지는 눈으로 다시 읽어서 판단하지 않는다. LLM 재검토는 "고쳐졌네요" 를 환각할 수 있으니까. 고치기 전/후로 같은 기계 검사(Next+MUI 면 번들 스캐너, 다른 스택이면 grep 재현)를 돌려 대상 신호가 사라졌는지(그리고 새 신호가 안 생겼는지)를 결정적으로 확인한다. 이건 린터로 코드를 고친 뒤 린트를 다시 돌리는 것과 같은 원리다. 그래서 6단계는 생략 불가다. 단 신호 규칙은 초안이라 오탐이 많으니(신호는 후보이지 판결이 아님), 히트 수 자체를 심각도로 쓰지 말고 오탐 필터를 거친다.

## 절차

### 0. 대상 스택 파악
대상이 이 레포/형제 Next+MUI 인지, 다른 스택(순수 HTML+CSS, Tailwind, Vue)인지 먼저 판별한다. 이게 이후 전역 스캔·회귀 검증을 번들 스캐너로 돌릴지 grep 재현으로 갈지 결정한다. 대상의 토큰 체계(테마 파일, CSS `:root`, tailwind config 등)가 어디 있는지도 찾는다.

### 1. 스코프 결정
- 단일 화면·컴포넌트: 그 화면(진입 파일 + 딸린 컴포넌트·스타일시트) 하나.
- 사이트 전수: 대상의 화면 인벤토리를 만들고 우선순위 = 공개 노출도 x 히트 밀도로 정렬한다. 노출도는 랜딩·주요 진입 페이지 최상, 로그인 뒤 중간, 어드민·도구 화면 최하(사용자 명시 없으면 제외 제안). 히트 밀도는 전역 기계 스캔으로 얻는다:
```bash
# 대상이 Next+MUI 일 때
node scripts/scan-slop-signals.mjs --json
node scripts/scan-layer-signals.mjs --json
# 다른 스택이면 각 신호 의도를 grep 으로 재현 (하드코딩 색, px 폰트, 100vh, 카피 중복 등)
```

### 2. 점검 디스패치 (slop-auditor, 병렬)
한 에이전트 = 한 화면으로 `slop-auditor` 를 디스패치한다. read-only 라 병렬이 안전하다. 하나의 메시지에서 여러 Task 를 동시 호출한다(run_in_background 금지, 이 프로젝트 병렬 규칙).

각 디스패치 프롬프트에 반드시 포함:
- 대상 스택과 토큰 체계 위치
- 화면 경로와 조립 파일 목록
- 해당 화면의 전역 스캔 히트 요약 (어디를 깊게 볼지)
- "기계 검사를 실제 실행(스캐너 또는 grep 재현)하고 오탐 필터를 통과시킬 것, 층별 리포트 형식 준수"

### 3. 집계
에이전트 리포트들을 모아서:
- **공용 컴포넌트 중복 통합**: 여러 화면이 같은 컴포넌트·스타일의 같은 발견을 보고하면 발견 1건 + 영향 화면 N개로 합친다. 수정도 1곳.
- **층 순서 정렬**: L0 → L1 → L2 → L3. 같은 층 안에서는 strong > 클러스터 승격 > weak 단독(보류 표기).
- **잠정 태그 보존**: 상류 의존 발견은 승인 표에서도 잠정으로 표기하고, 상류 수정 후 재점검 대상임을 명시한다.
- **기계 검증 / 2차 소견 구분 유지**: 소견만으로 강행 수정하지 않는다. 소견 발견은 사용자가 화면에서 직접 확인하도록 근거를 제시한다.

### 4. 승인 게이트 (유일한 필수 정지 지점)
발견별로 표를 제시하고 어떤 항목을 수정할지 승인받는다:

| 층 | 발견 | 근거 유형 | 영향 파일/화면 | 수정안 [삭제/축소/교체] | 위험도 |
|---|---|---|---|---|---|

- 수정안에는 처방 사다리 단계를 명시한다. **삭제·축소가 기본**이고, 교체는 그 슬롯이 반드시 남아야 할 때만.
- 위험도: 카피 변경(사용자 노출 텍스트) > 구조 변경(마크업 재배치·요소 삭제) > 스타일 변경(토큰 교체).
- 단일 화면 스코프면 이 표 하나가 유일한 세리머니다. 표 승인 즉시 과감하게 적용하고, 전수 스캔·인벤토리 같은 사이트 전수용 단계를 끌어오지 않는다.

### 5. 승인 항목 수정 (층 순서, 순차)
- **L0 부터.** 콘텐츠 구조 수정(중복 요소 제거, 상수화, 위계 재정렬)을 먼저 적용한다.
- L0 적용 후, 잠정 태그가 붙었던 하류 발견을 재확인한다. 요소가 사라졌으면 발견 폐기, 남아 있으면 그대로 진행. 대규모면 해당 화면만 slop-auditor 재디스패치.
- L1 → L2 → L3 순서로 나머지를 적용한다. 같은 파일을 동시에 만지지 않도록 파일 단위로 순차 진행(이 프로젝트에서 병렬 쓰기는 불안정). 공용 컴포넌트·스타일 수정이 단독 파일 수정보다 먼저.
- 모든 수정은 최소 diff. 무관한 줄을 건드리지 않는다. 상수화·토큰 승격은 대상 프로젝트의 관례(상수 파일, CSS `:root`, tailwind config 등)를 따른다. 대상이 이 레포면 `src/data/contents.js` + content-inventory 규약.

### 6. 회귀 검증
```bash
# 대상이 Next+MUI 일 때
node scripts/scan-slop-signals.mjs <수정된 경로들>
node scripts/scan-layer-signals.mjs <수정된 경로들>
# 다른 스택이면 5단계에서 쓴 grep 재현을 수정 경로에 다시 실행
```
수정 대상 신호가 사라졌는지, 수정이 새 신호를 만들지 않았는지 확인한다. 수정 전 히트맵과 대비해 보고한다. 사이트 전수였다면 전역 스캔도 한 번 더.

## 최종 보고 형식

```
## 슬롭 수정: <스코프> <날짜>
점검 화면 <R>개 · 발견 <F>건 (승인 <A> / 보류 <H> / 상류 수정으로 폐기 <D>)

### 수정 완료 (층 순서)
| 층 | 발견 | 영향 화면 | 수정 내용 (처방 출처) | 재검증 |
|---|---|---|---|---|

### 보류
| 발견 | 사유 (미승인 / weak 단독 / 소견 단독) |

### 커버리지
기계 검사(스캐너 또는 grep 재현) 실행 · 무히트 신호 <n>개 · 층별 판단 확인 완료

### 미등재 관찰
<택소노미 추가 후보. 발견과 분리>
```

## 흔한 함정

- **하류부터 고치지 말 것.** L2 타이포를 먼저 고친 뒤 L0 에서 그 요소가 삭제되면 작업이 증발한다. 층 순서는 미관이 아니라 수정 무효화 방지 장치다.
- **히트 수를 심각도로 쓰지 말 것.** 신호는 오탐 조건을 통과해야 발견이다.
- **소견을 검증처럼 팔지 말 것.** 위계가 읽히는지, 밀도가 과한지는 2차 소견이다. 사용자 확인 없이 소견만으로 카피·구조를 바꾸지 않는다.
- **대상 관례를 이 레포 기준으로 덮어쓰지 말 것.** lucide·클로드 오렌지·프레젠테이션 다크 테마·text.disabled 금지는 *이 레포* 특수성이다. 대상이 다른 프로젝트면 그 프로젝트의 토큰·관례를 존중하고, 그 스택의 idiom(순수 CSS 의 hex 등)을 결함으로 몰지 않는다.
- **어드민·도구 화면에 공개 페이지 기준을 강요하지 말 것.**
- **경계**: 장문 한글 윤문은 humanize-korean, 성능은 frontend-perf-auditor, 깊은 레이아웃 안정성 감사는 stable-layout-auditor, 타이포 양성 준수 상세는 typography-auditor.
