# Slop Fix

대중이 느끼는 "AI 티"를 지우는 작업 스킬. 역할 분리가 핵심이다:

- **점검 = `slop-auditor` 에이전트** (read-only, 독립 컨텍스트). 게슈탈트 판정 + Top 3~5 과감 수정안을 낸다. 생성한 쪽이 자기 검증하지 않는다.
- **수정 = 이 스킬** (메인 컨텍스트에서 Edit). 승인받은 Top 수정안을 순서대로 적용한다.

## 핵심 원칙

- **가성비 우선, Top 3~5 만.** 산출물은 화면이 덜 AI스러워 보이는 소수의 과감한 수정이다. 17건짜리 전수 리포트, 디자이너급 디테일 폴리싱은 이 스킬의 실패 모드다.
- **과감하게, 빼는 쪽으로.** 처방 사다리는 삭제 > 축소 > 교체. 기믹 위젯 삭제, 카피 축소, 섹션 통합이 기본기고, 색만 바꾸는 소심한 재스타일링은 금지. 위험은 승인 게이트가 막아준다.
- **위생은 스코프 밖.** px→rem, 토큰 정리, a11y 가드 같은 화면 안 바뀌는 수정은 사용자가 명시 요청할 때만. 위생만 고치고 화면이 그대로면 이 스킬은 실패한 것이다. (위생 전용 스캐너 scan-layer-signals.mjs 도 그때만 쓴다.)
- **처방 출처는 SSOT, 런타임 로드.** 무엇이 슬롭이고 무엇으로 탈출하는지는 `src/data/aiSlopTaxonomyData.js` 의 tell/escape/escapeNote 에서 온다. 항목 수·버전을 가정하지 않는다. 택소노미가 자라면 이 스킬의 수비 범위도 자동으로 자란다. 표현은 대상 프로젝트 자신의 토큰·관례로 쓴다.
- **수정 순서는 의존성 순서.** 콘텐츠·카피 → 구조 → 타이포 → 표면. 상류 수정이 하류 발견을 무효화할 수 있으니 `잠정` 표기 발견은 상류 적용 후 재확인한다.
- **수정은 반드시 사용자 승인 후.** 점검은 자율, 쓰기는 게이트 뒤. em-dash(U+2014) 0건.

## 절차

### 1. 점검 디스패치
대상 스택과 토큰 체계 위치를 파악해 `slop-auditor` 에게 한 화면 스코프로 넘긴다. 프롬프트에 반드시: 대상 경로·스택·토큰 위치, "게슈탈트 판정 먼저, 스캐너는 판정 뒤 조준·회귀용만, Top 3~5 형식 준수".

### 2. 승인 게이트 (유일한 정지 지점)
에이전트의 Top 수정안을 표로 승인받는다. 단일 화면이면 이 표 하나가 유일한 세리머니다.

| # | 텔 (SSOT id) | **[삭제/축소/교체]** | 영향 파일 | 기대효과 | 위험도 |
|---|---|---|---|---|---|

위험도: 카피 변경 > 구조 변경·요소 삭제 > 스타일 교체.

### 3. 버전 보존 후 적용
- 대상이 사용자의 버전관리 밖이거나 원본 보존을 원하면: `<대상>-deslop` 으로 복사 → `git init` → baseline 커밋 → **수정 1건 = 커밋 1개** → `CHANGELOG-deslop.md` 에 판정·사다리·before/after 기록. (이미 git 관리 중이면 브랜치로 동일하게.)
- 승인 항목을 의존성 순서(상류부터)로 적용한다. 최소 diff 가 아니라 **과감한 diff**: 삭제 대상은 관련 CSS·JS 까지 함께 걷어낸다. 단 승인 안 받은 요소는 건드리지 않는다.
- 상류 적용 후 `잠정` 발견을 재확인해 소멸했으면 폐기한다.

### 4. 회귀 확인
- 코드-텔을 고쳤으면 `node scripts/scan-slop-signals.mjs <수정 경로>` 재실행(비 JS 스택이면 같은 신호의 grep 재현)으로 신호 소멸을 증명한다.
- 렌더 가능하면 before/after 스크린샷 1쌍으로 체감 변화를 보여준다. 이 스킬의 성공 기준은 "눈으로 달라 보임"이다.

## 흔한 함정

- **위생으로 분량 채우지 말 것.** 화면이 안 바뀌는 수정을 성과로 팔면 이 스킬의 존재 이유가 죽는다.
- **하류부터 고치지 말 것.** 카피·요소 삭제가 확정되기 전에 그 요소의 타이포·색을 다듬으면 작업이 증발한다.
- **소심한 교체로 도망가지 말 것.** 삭제가 가능한 자리에서 교체를 고르면 안 된다. 사다리는 위에서부터.
- **대상 관례를 홈 레포 기준으로 덮어쓰지 말 것.** 다른 프로젝트의 idiom(순수 CSS hex 등)은 결함이 아니다.
- **경계**: 장문 한글 윤문 humanize-korean, 성능 frontend-perf-auditor, 레이아웃 안정성 stable-layout-auditor, 타이포 상세 typography-auditor.

## 별첨: 사이트 전수 모드 (명시 요청 시만)

여러 화면을 훑어야 할 때만: 화면 인벤토리 → 공개 노출도순 정렬 → slop-auditor 를 화면당 1개씩 병렬 디스패치(read-only 라 안전, 하나의 메시지에서 동시 Task 호출, run_in_background 금지) → 공용 컴포넌트 중복 통합 → 단일 승인 표 → 위 3~4단계와 동일. 기본값은 어디까지나 단일 화면이다.
